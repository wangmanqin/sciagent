# SciAgent 技术深度解析（面试深度回答用）

这份文档逐文件、逐逻辑地解释整个项目是怎么搭建的。
读完后你能回答"你是怎么搭建的"这个问题，而且是从底层讲起。

---

## 整体架构：谁调用谁？

```
用户 ──→ app.py（网页界面）
              │
              ↓
         agent.py（主循环）←──── prompts.py（系统指令）
              │
         ┌────┴────┐
         ↓         ↓
      llm.py    tools.py
     (调API)   (执行代码)
         │
         ↓
    DeepSeek 服务器
```

调用顺序：用户在网页输入 → app.py 接收 → 调用 agent.py → agent.py 反复调用 llm.py 和 tools.py → 最终返回结果给 app.py → 网页展示

下面从底层往上逐层讲解。

---

## 第一层：tools.py（最底层，代码执行器）

### 这个文件解决什么问题？

大模型生成了一段代码字符串，比如 "print(1+1)"。
但字符串不是程序——你需要一个东西真的去运行它。
tools.py 就是那个"运行器"。

### 核心逻辑（伪代码）：

```
函数 run_python_code(code字符串):

    第1步：记录现在 outputs 文件夹里有哪些 .png 图片
           → 目的：等会儿用来判断"这次运行新生成了哪些图"

    第2步：把 code 字符串写入一个临时文件 _temp_script.py
           → 为什么不直接运行字符串？因为操作系统只能运行文件，不能运行字符串

    第3步：调用操作系统命令 "python _temp_script.py"
           → 这跟你在终端里手动输入 "python xxx.py" 是一样的
           → 设置了 60 秒超时，防止死循环卡死

    第4步：捕获运行结果
           → 如果正常运行：拿到 print 输出的内容（stdout）
           → 如果报错了：拿到错误信息（stderr）
           → 如果超时了：返回 "超时" 错误

    第5步：再看一次 outputs 文件夹里的 .png 图片
           → 新增的图片 = 现在的图片 - 第1步记录的图片
           → 如果有新图片，把文件名也加到返回结果里

    第6步：把所有信息拼成一个字符串返回
```

### 另一个重要的东西：TOOL_DEFINITIONS（工具说明书）

```
TOOL_DEFINITIONS = [
    {
        名称: "run_python_code",
        描述: "能执行 Python 代码，环境里装了 numpy、scipy、matplotlib、deap",
        参数: {
            code: 字符串类型，"要执行的完整 Python 代码"
        }
    }
]
```

这个说明书不是给人看的，是给大模型看的。
我们调 API 时会把这个说明书发过去，大模型看到后就知道它可以用这个工具。

### 还有一个：TOOL_EXECUTORS（工具执行映射表）

```
TOOL_EXECUTORS = {
    "run_python_code" → 调用 run_python_code 函数
}
```

为什么需要这个？因为 agent.py 收到大模型的指令时，指令里只有一个字符串 "run_python_code"。
agent.py 需要一个映射表来查找：这个名字对应的是哪个函数？然后才能调用。

如果以后你加新工具（比如 "read_csv"），只需要：
1. 写一个新函数
2. 在 TOOL_DEFINITIONS 里加一条说明
3. 在 TOOL_EXECUTORS 里加一条映射
Agent 就自动获得了新能力。这就是工具扩展机制。

## 第二层：llm.py（API 通信层）

### 这个文件解决什么问题？

agent.py 需要和大模型对话，但不同的大模型 API 格式不一样。
llm.py 把这些差异屏蔽掉，对外提供统一的接口。

### 统一的数据结构

不管用哪个大模型，llm.py 返回的结果格式都是一样的：

```
LLMResponse:
    text:        大模型说的话（字符串）
    tool_calls:  大模型想调用的工具（列表，可能为空）
    has_tool_calls: 是否有工具调用（true/false）

ToolCall:
    id:          这次调用的唯一标识（用来把结果和调用配对）
    name:        工具名称，比如 "run_python_code"
    arguments:   工具参数，比如 {"code": "print(1+1)"}
```

### DeepSeekLLM 的核心逻辑（伪代码）

```
函数 chat(消息列表, 工具清单):

    第1步：格式转换
           我们程序内部用的是一种统一格式
           DeepSeek API 要求的是 OpenAI 兼容格式
           这一步做的就是格式转换（就像把 Word 文档转成 PDF）

           重点转换：
           - 工具说明书：我们的格式 → OpenAI 的 function 格式
           - 历史消息：尤其是工具调用结果，OpenAI 要求用特殊的 "tool" 角色

    第2步：发送 HTTP 请求给 DeepSeek 服务器
           → 包含：模型名称、所有历史消息、工具清单、最大输出长度
           → DeepSeek 服务器处理后返回响应

    第3步：解析响应
           DeepSeek 返回的内容可能包含：
           情况A：只有文字 → 说明大模型不想用工具，直接给了回答
           情况B：有文字 + 工具调用 → 大模型说了一些话，同时请求调用工具
           情况C：只有工具调用 → 大模型不说话，直接请求调用工具

           对于工具调用，需要解析出：
           - 工具名称（比如 "run_python_code"）
           - 参数（比如那段代码字符串）
           注意：DeepSeek 返回的参数是 JSON 字符串，需要解析成数据结构
                 但 DeepSeek 有时返回的 JSON 不合法，所以加了容错处理

    第4步：把解析结果包装成统一的 LLMResponse 格式返回
```

### 为什么要做"格式转换"？

大模型 API 的消息格式是一个严格的协议。以一轮工具调用为例：

```
我们发给 API 的消息列表：

消息1 [system]:  "你是科学计算专家..."
消息2 [user]:    "计算素数"
消息3 [assistant]: "我来写代码" + 工具调用请求{run_python_code, code="..."}
消息4 [tool]:    "执行结果：2,3,5,7,11..."

API 返回：
消息5 [assistant]: "100以内共有25个素数..."（最终回答）
```

每一轮对话，完整的历史消息都要发过去。大模型是"无记忆"的，它不记得之前说了什么，
所以每次调用都要把所有历史消息带上。这也是为什么 agent.py 要维护一个消息列表。

### 三种模式的区别

- MockLLM: 不联网，返回预设的固定回答。开发调试用的。
- DeepSeekLLM: 调 DeepSeek 的 API。国内可用，用 OpenAI 兼容格式。
- ClaudeLLM: 调 Claude 的 API。格式略有不同（用 Anthropic 自己的 SDK）。

create_llm 函数会自动检测你配置了哪个 API Key，选择对应的模式。

## 第三层：agent.py（核心！ReAct 主循环）

### 这个文件解决什么问题？

它是"项目经理"，负责协调 llm.py 和 tools.py，实现 ReAct 循环。

### run 函数的完整逻辑（伪代码）

```
函数 run(用户输入):

    第1步：初始化消息列表
           消息列表 = [
               {角色: system, 内容: System Prompt},  ← 来自 prompts.py
               {角色: user, 内容: 用户输入的问题}
           ]

    第2步：开始循环（最多 10 轮）

        第2.1步：调用大模型
                 把 消息列表 + 工具清单 发给 llm.py
                 拿到返回结果 response

        第2.2步：判断返回结果
                 如果 response 里没有工具调用：
                     → 大模型直接给了文字回答，任务完成
                     → 返回这段文字，循环结束

                 如果 response 里有工具调用：
                     → 大模型想用工具，继续往下走

        第2.3步：记录大模型的回复
                 把大模型说的话 + 它请求的工具调用，作为一条 assistant 消息
                 追加到消息列表
                 → 为什么？因为下次调 API 要带上完整历史

        第2.4步：执行工具
                 对于大模型请求的每个工具调用：
                     从 TOOL_EXECUTORS 映射表查到对应的函数
                     调用函数，拿到执行结果（或错误信息）

        第2.5步：记录工具执行结果
                 把每个工具的执行结果，作为 tool_result 消息
                 追加到消息列表
                 → 下一轮循环时，大模型会看到这些结果

        第2.6步：回到第 2.1 步，开始下一轮
                 大模型这次会看到完整的历史：
                 原始问题 → 它上次的回复 → 工具执行结果
                 然后它决定：继续调工具？还是给出最终回答？

    第3步：如果循环了 10 轮还没结束，强制停止
           → 防止大模型陷入无限循环（比如一直报错一直重试）
```

### 一个具体例子：消息列表的变化过程

用户输入："计算100以内的素数并画图"

```
═══ 初始状态 ═══
消息列表 = [
    {system: "你是科学计算专家..."},
    {user: "计算100以内的素数并画图"}
]

═══ 第1轮：调 API ═══
→ 发送上面2条消息 + 工具清单
← 大模型返回：文字"我来计算" + 工具调用{run_python_code, code="...计算素数..."}

消息列表变为 = [
    {system: "你是科学计算专家..."},
    {user: "计算100以内的素数并画图"},
    {assistant: "我来计算" + 工具调用{run_python_code}},   ← 新增
    {tool_result: "2,3,5,7,11... 生成了 primes.png"}       ← 新增（执行结果）
]

═══ 第2轮：调 API ═══
→ 发送上面4条消息 + 工具清单
← 大模型看到"执行成功"，返回：纯文字"100以内共有25个素数..."

消息列表变为 = [
    {system: "你是科学计算专家..."},
    {user: "计算100以内的素数并画图"},
    {assistant: "我来计算" + 工具调用},
    {tool_result: "2,3,5,7,11..."},
    {assistant: "100以内共有25个素数..."}   ← 新增（最终回答）
]

→ 大模型没有请求工具调用 → 循环结束 → 返回最终回答
```

### Self-Correction 是怎么发生的？

```
═══ 第1轮 ═══
大模型写了一段代码，但忘记了 import numpy
→ 工具执行报错："ModuleNotFoundError: No module named 'numpy'"
→ 错误信息被追加到消息列表

═══ 第2轮 ═══
大模型看到消息列表里的错误信息
→ 它自己分析："哦，我忘记 import 了"
→ 生成修正后的代码（加上了 import numpy）
→ 再次请求执行
→ 这次成功了

没有任何特殊的"自我修正算法"。
就是把错误信息当作新的输入发回去，大模型自然会尝试修正。
```

### on_event 回调是什么？

agent.py 在运行过程中，每发生一件事（开始思考、调用工具、得到结果、出错、最终回答），
都会触发一个"事件"。app.py 通过 on_event 回调函数接收这些事件，实时显示在网页上。

这就是为什么你在网页上能看到 Agent 的思考过程——不是事后展示的，是实时推送的。

## 第四层：prompts.py（System Prompt）

### 这个文件很短，但极其重要

它就是一大段英文文本，告诉大模型该怎么行为。关键的设计决策：

1. **规定工作流程**：分析 → 写代码 → 执行 → 调试 → 可视化 → 解释
   → 如果不规定，大模型可能跳步骤，比如直接给出不准确的口算结果

2. **规定代码规范**：必须 print、必须 savefig、不准用 plt.show()
   → 如果不说"必须 print"，执行结果就是空的，大模型拿不到数据
   → plt.show() 会弹窗但服务器没有屏幕，会卡住

3. **规定图表语言**：所有图表文字必须用英文
   → 因为 matplotlib 不支持中文字体，中文会显示方块

4. **规定重试上限**：最多 5 次
   → 防止大模型陷入"改了还是错"的死循环

每一条规则都是从测试中发现问题后加上去的。
这就是 Prompt Engineering 的本质：发现问题 → 加约束 → 测试 → 循环。

## 第五层：app.py（Streamlit 前端）

### 核心逻辑

```
1. 页面加载时，从 history/chat_history.json 读取历史记录并显示
2. 用户在输入框打字并回车
3. 创建一个 Agent 实例
4. 调用 agent.run()，同时传入 on_event 回调
5. on_event 每次被触发，就在网页的 "Agent 正在工作" 面板里新增一行
6. agent.run() 返回后，显示最终回答和生成的图片
7. 把本次对话保存到 chat_history.json
```

## 导师问"你怎么搭建的"，你可以这样回答

> 整个系统分四层：
>
> 最底层是工具层，负责在沙箱环境中执行 Python 代码，捕获标准输出和错误输出。
>
> 第二层是 API 通信层，封装了 DeepSeek API 的调用，处理消息格式转换和
> Function Calling 的请求解析。这里有一个设计决策：我做了统一的响应格式，
> 这样切换不同的大模型只需要换一个类，上层代码完全不用改。
>
> 核心是第三层，Agent 主循环。它维护一个消息列表，每一轮把完整的对话历史
> 和工具清单发给大模型，大模型返回后判断它是给出最终回答还是请求调用工具。
> 如果是工具调用，就执行工具，把结果追加到消息列表，进入下一轮。
> 这就是 ReAct 循环的实现。Self-correction 也是在这个循环里自然发生的——
> 代码报错的信息作为 tool_result 反馈给大模型，大模型在下一轮自行修正。
>
> 还有一个关键部分是 System Prompt 的设计。我通过反复测试发现了几个问题，
> 比如大模型生成的代码不输出结果、画图时中文乱码、代码缺少必要的 import，
> 都是通过在 Prompt 中加入明确的约束来解决的。这个过程让我深刻理解了
> Prompt Engineering 不是随便写几句话，而是一个发现问题、加约束、验证的迭代过程。

## 设计决策总结（面试高分回答素材）

| 决策 | 为什么这样做 |
|------|-------------|
| 工具执行用 subprocess 而不是直接 exec | 安全隔离。subprocess 在独立进程中执行，主程序不受影响。exec 直接在当前进程执行，恶意代码可能破坏主程序 |
| 设置 60 秒超时 | 防止大模型生成死循环代码导致程序卡死 |
| 最多循环 10 轮 | 防止大模型陷入无限的 "报错→修复→又报错" 循环 |
| 统一 LLMResponse 格式 | 解耦。切换大模型（DeepSeek→Claude）只需换一个类，agent.py 完全不用改 |
| 消息列表累积而不是只发最新一条 | 大模型是无记忆的。它不记得之前说了什么，必须每次把完整历史带上 |
| 工具结果区分 is_error | 让大模型能区分"执行成功"和"执行报错"，做出不同的响应 |
| Prompt 中强制用英文画图 | matplotlib 默认不支持中文字体，中文会显示方块。这是通过测试发现的问题 |
| 用列表保存 on_event 事件 | 前端可以实时展示 Agent 的每一步思考和行动过程 |
